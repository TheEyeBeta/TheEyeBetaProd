# doc-sync

This repo's docs went stale for months because nothing forced them to track code changes — three
different port maps disagreed with each other, the DB schema description was pure fiction, and
the agent architecture doc described a system that was never built. The 2026-06-18 rewrite fixed
that once. This skill's job is to stop it from happening again, one change at a time.

## What this skill does NOT do

- It does not touch `CHANGELOG.md` — that's generated by `git-cliff` from Conventional Commit
  messages (`cliff.toml`), already automatic.
- It does not write ADRs — architectural decisions need a human to fill in Context/Decision/
  Consequences/Alternatives (`docs/templates/adr.md`); this skill should flag when one looks
  warranted, not draft one.
- It does not invent facts. If the diff doesn't make the right answer obvious, say what's unclear
  and ask, rather than guessing a port number or table name.

## Procedure

### 1. Determine the diff scope

```bash
git diff main...HEAD --name-only   # preferred: full branch diff
# fall back to working-tree/staged changes if there's no upstream diff yet:
git diff --name-only; git diff --cached --name-only
```

Filter to the path patterns in the lookup table below. If nothing matches, say so and stop — don't
touch unrelated docs.

### 2. Classify each changed path and find its target doc(s)

| Changed path pattern | What probably changed | Update these docs |
|---|---|---|
| New/removed directory under `services/<name>/` | A service was added, removed, or its purpose changed | `docs/architecture.md` §2.2 (port map) + §3.1 (service inventory); `README.md` Service Map; `docs/repo-layout.md` tree; **add/update a row in `SERVICES_STATUS.md`** with real deploy status — don't mark it deployed unless you can point to a systemd unit or compose entry |
| New file under `db/migrations/versions/` | A schema change | `docs/architecture.md` §4.1 domain table (add the migration to the right domain row, or add a new row); `docs/data-model.md` if it introduces a new table family, hypertable, or role worth calling out — skip purely additive/column-only migrations unless they change a documented invariant |
| Edits to `docker-compose.yml` (service add/remove/port/healthcheck change) | Dev-infra topology changed | `docs/architecture.md` §2.2; `README.md` Service Map; `docs/repo-layout.md` |
| New/changed file under `deploy/systemd/` (not `archived/` or `staged/`) | A service went from code-complete to actually deployed, or vice versa | `SERVICES_STATUS.md` (move the row to "deployed" or back); `docs/architecture.md` §2.2 Runtime column and §3.1 |
| File moved into/out of `deploy/systemd/staged/` or `archived/` | Deploy intent changed without going live | `SERVICES_STATUS.md` notes column |
| Edits to `config/agents/hierarchy.yaml` or `config/litellm.yaml` | Agent roster or model routing changed | `docs/agents.md` — update the department table or the model-routing paragraph; do NOT copy the full agent list into the doc, it's explicitly designed to stay thin and point at the YAML |
| New/changed file under `.github/workflows/` | CI/CD topology changed | `docs/ci.md` — update the job graph/table for the affected workflow |
| New file under `docs/adr/` | A new ADR was added | Check `ls docs/adr/ | tail -1` for the next free number — if the new file's number collides with an existing one, that's a bug, flag it immediately (this exact collision happened twice before: 0001 and 0002 were each claimed by two unrelated ADRs) |

### 3. Read, draft, apply

For each target doc: read the current section (don't skim — these docs have been wrong before in
ways that looked plausible), draft the specific edit grounded in what the diff actually shows, and
apply it with the Edit tool. Keep edits surgical — match the existing structure/section numbering,
don't restructure a doc while fixing one fact.

### 4. Summarize

End with:
- A short list of what was changed, per file.
- Anything that looks like it crossed from "code change" into "architectural decision" (new
  service topology, new schema design, new security model) — suggest an ADR rather than writing
  one.
- Anything you couldn't resolve confidently from the diff alone (e.g. "is this service actually
  deployed, or just code-complete?") — ask rather than guess; `SERVICES_STATUS.md` has burned
  readers before by claiming things were live that weren't.
